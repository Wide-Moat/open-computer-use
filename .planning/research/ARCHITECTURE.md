## Recommendations (answering the 8 questions)

### 1. Adapter location → **(b) `computer-use-server/cli_runtime.py` + `cli_adapters/` package**

- **Not (a) inline:** `mcp_tools.sub_agent` is already 318 lines with three failure-mode branches (kill, timeout, success); embedding three more builders bloats it past comprehensibility and makes unit testing require spinning up FastMCP. Keep `sub_agent` as orchestration, push string-building down a layer.
- **Not (c) in-container wrapper script:** A `/usr/local/bin/sub-agent-cli` would have to receive `task`, `model`, `system_prompt` through argv or env, which means re-quoting twice (Python → shell → wrapper → CLI). It also splits the codebase across language boundaries (bash adapter) and makes adding env passthrough require both a Python and a bash change. The orchestrator already builds CLI commands (line 1012); keep it there.
- **Pick (b):** Pure-Python module(s), no Docker dependency, fast unit tests, single commit changes one layer. The adapter receives `(task, system_prompt, model, max_turns, working_directory, resume_session_id, user_email)` and returns a `(command_string, result_parser)` tuple.

### 2. Resolver flow → **`SUBAGENT_CLI` is read once at `docker_manager.py` module load (alongside `CLAUDE_CODE_PASSTHROUGH_ENVS` at line 94), passed through to every container as an env var, and re-read at sub_agent invocation time from the orchestrator's process env.**

Justification:
- Reading at orchestrator boot mirrors the existing pattern for every other knob (`SUB_AGENT_DEFAULT_MODEL`, `ANTHROPIC_BASE_URL`, etc., lines 70–91 of `docker_manager.py`) — no new pattern.
- Propagating into the container is required regardless: the ttyd terminal autostart (`.bashrc` line 395) needs to know which CLI to launch, and that runs inside the sandbox.
- Re-reading at invocation time (single `os.getenv` lookup) is free and lets a single orchestrator support config reloads via restart without rebuilding the image.
- Per-call override is explicitly out-of-scope (PROJECT.md decision), so there's no need to read `SUBAGENT_CLI` from request headers / context vars.

### 3. Env passthrough → **Three CLI-scoped tuples + one union iteration site.**

```python
# docker_manager.py
CLAUDE_CODE_PASSTHROUGH_ENVS = (...)   # existing, line 94
CODEX_PASSTHROUGH_ENVS = (
    ("OPENAI_API_KEY", os.getenv("OPENAI_API_KEY", "")),
    ("OPENAI_BASE_URL", os.getenv("OPENAI_BASE_URL", "")),
    ("CODEX_MODEL", os.getenv("CODEX_MODEL", "")),
)
OPENCODE_PASSTHROUGH_ENVS = (
    ("OPENROUTER_API_KEY", os.getenv("OPENROUTER_API_KEY", "")),
    ("OPENCODE_MODEL", os.getenv("OPENCODE_MODEL", "")),
    ("OPENCODE_PROVIDER", os.getenv("OPENCODE_PROVIDER", "")),
)
ALL_CLI_PASSTHROUGH_ENVS = (
    CLAUDE_CODE_PASSTHROUGH_ENVS
    + CODEX_PASSTHROUGH_ENVS
    + OPENCODE_PASSTHROUGH_ENVS
)
```

Reason: tests can `from docker_manager import CODEX_PASSTHROUGH_ENVS` and assert exactly the codex-only contract without filtering a giant union, but the `_create_container` loop at line 461 iterates `ALL_CLI_PASSTHROUGH_ENVS` once. Readable for humans, cheap for tests, matches the GATEWAY-03 pattern (deterministic tuple iteration).

### 4. Model resolution → **One dispatch function with three private resolvers.**

```python
# cli_runtime.py
def resolve_subagent_model(alias_or_id: str, cli: str) -> tuple[str, str]:
    """Returns (model_id_for_cli, display_name)."""
    cli = cli or "claude"
    return _RESOLVERS[cli](alias_or_id)

_RESOLVERS = {
    "claude":   _resolve_claude,
    "codex":    _resolve_codex,
    "opencode": _resolve_opencode,
}
```

Justification: keeps a single import surface for `mcp_tools` (it already uses one `ALIAS_MAP` today, lines 909–913). Each `_resolve_*` is private and tiny, and a new CLI can be added by registering one new entry. Three independent public functions would force `sub_agent` to grow its own dispatch — that responsibility belongs to the runtime module, not the MCP tool.

### 5. Sub-agent skill prompts → **Keep a shared skill, parametrise the runtime-specific quirks.**

The `skills/public/sub-agent/SKILL.md` content is _instructional_ (when to use, ROLE/DIRECTIVE/CONSTRAINTS template) and _prescriptive_ (resume semantics, max_turns guide). None of that is Claude-specific.

What _is_ Claude-specific today and must move:
- The `--resume <session_id>` advice (line 137–140): codex and opencode don't share that flag. Move to a new `references/runtimes.md` with a per-CLI table.
- The session log path `~/.claude/projects/-home-assistant/<session-id>.jsonl` (line 124): claude-only. Move to the same per-CLI reference.

The system_prompt embedded in `mcp_tools.sub_agent` (lines 980–1005) is also runtime-agnostic — it talks about `/mnt/user-data/uploads`, skills, plan files. Pass it to all three adapters unchanged. Each adapter decides how to inject it (`claude --append-system-prompt`, `codex` reads from a file or arg, `opencode` reads `--system`).

Decision: **one skill file, one per-runtime reference**. No fork.

### 6. ttyd terminal UX → **Keep `tmux ... bash` as the ttyd command (no change to `app.py:848`); change only the `.bashrc` autostart line at `Dockerfile:395` so it dispatches on `$SUBAGENT_CLI`.**

```bash
# .bashrc autostart (replacement for line 395)
[ -z "$SUBAGENT_AUTOSTARTED" ] && [ -n "$PS1" ] && export SUBAGENT_AUTOSTARTED=1 \
    && exec "${SUBAGENT_CLI:-claude}"
```

Justification:
- ttyd already launches `tmux ... bash`, and bash auto-execs the chosen CLI on first interactive login. That gives the user the experience they want (open terminal → CLI prompt) without changing the terminal proxy plumbing in `app.py`.
- Using `exec` (not just `claude` as a foreground call) means Ctrl-D from the CLI ends the tmux pane cleanly — same UX as today.
- Falling back to `claude` keeps backwards-compat for any deployment that hasn't set `SUBAGENT_CLI`.
- We rename the marker variable from `CLAUDE_AUTOSTARTED` to `SUBAGENT_AUTOSTARTED` to avoid confusion when the runtime isn't claude. This is a tiny migration: existing volumes will see no `CLAUDE_AUTOSTARTED` flag, the new line still picks up `SUBAGENT_AUTOSTARTED`, autostart fires once. No data loss.

### 8. OpenCode config file lifecycle → **Generate `~/.config/opencode/opencode.json` in the entrypoint script when `$SUBAGENT_CLI=opencode`, from env vars (`OPENCODE_PROVIDER`, `OPENCODE_MODEL`, `OPENROUTER_API_KEY`, etc.).**

- **Not bake into image:** would hard-code provider, defeats the "operator brings own keys" rule (PROJECT.md out-of-scope #3).
- **Not mount from host:** new mount, new docker-compose surface, leaks operator state into the codebase. Already rejected for Anthropic auth (handled via env).
- **Generate at entrypoint:** matches the existing pattern at `Dockerfile:316–352` (CLAUDE.md, settings.json, .claude.json all written by entrypoint). Same locality, same lifecycle (per-container fresh write on every boot), no secrets in image, no host mount.

## Data Flow (env-switch → CLI invocation → result capture)

```
operator sets SUBAGENT_CLI=codex in .env
        ↓
docker-compose up
        ↓
computer-use-server starts → docker_manager.py imports SUBAGENT_CLI at module load
        ↓
chat starts → MCP /sub_agent call arrives
        ↓
mcp_tools.sub_agent(task, model="sonnet")
        ↓
cli_runtime.resolve_subagent_model("sonnet", cli="codex")
        → ("gpt-5-codex", "sonnet→gpt-5-codex")
        ↓
cli_runtime.build_command(cli="codex", task, system_prompt, model_id, ...)
        → "cd /home/assistant && OPENAI_API_KEY=... codex exec --model gpt-5-codex --json --max-turns 25 ..."
        ↓
docker_manager._execute_bash(container, command, SUB_AGENT_TIMEOUT)
        ↓
codex stdout/stderr captured → cli_adapters.codex.parse_result(stdout)
        → {"text": ..., "turns": N, "cost": $, "session_id": ..., "is_error": bool}
        ↓
mcp_tools._format_sub_agent_result(parsed) → MCP response to caller
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Per-call CLI override

**What people do:** Add `cli="codex"` to the `sub_agent(...)` signature.
**Why it's wrong:** Multiplies test surface 3×; skill docs would have to teach three call patterns; race condition between the runtime baked into the container env (for the ttyd terminal) and the per-call choice.
**Do this instead:** Boot-time only (PROJECT.md decision). Operator picks one CLI per deployment.

### Anti-Pattern 2: Reading `SUBAGENT_CLI` from request headers

**What people do:** Plumb `X-Subagent-Cli` through `set_context_from_headers`.
**Why it's wrong:** Same as #1 plus security: the chosen CLI determines which env vars are passed in (OPENAI_API_KEY vs ANTHROPIC_AUTH_TOKEN), and a header would let any caller pivot the orchestrator's secret surface.
**Do this instead:** Pure env var, set at orchestrator boot, never overridden by request.

### Anti-Pattern 3: Three independent `claude_command`-style strings inside `mcp_tools.sub_agent`

**What people do:** Three `if SUBAGENT_CLI == "..."` branches each with its own multi-line shell-quoting block.
**Why it's wrong:** `sub_agent` is already 318 lines with intricate timeout / kill / log-streaming logic. Tripling the command-build section pushes it past 500 and entangles dispatch with orchestration.
**Do this instead:** Adapter pattern (recommendation #1). One call site in `sub_agent`, three small builders unit-tested independently.

### Anti-Pattern 4: Generating `opencode.json` from Python before `docker exec`

**What people do:** Build the JSON in `_create_container` and `put_archive` it into the container.
**Why it's wrong:** Contains a secret (`OPENROUTER_API_KEY`); two write paths (entrypoint for ttyd, Python for sub_agent) drift; entrypoint already has the right primitives.
**Do this instead:** Single source of truth — entrypoint writes the file from container env (recommendation #8). Python only sets env vars on the container.
