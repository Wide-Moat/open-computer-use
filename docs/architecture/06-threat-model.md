STRIDE-per-element threat model over the Layer 6 container DFD, with each threat resolved to a controlling NFR. Audience: security engineers and auditors reviewing the trust-boundary controls.

## 1. Scope and DFD reference

This model runs STRIDE-per-element over the containers, one data store, the external actors, and the boundary-crossing flows fixed in [`05-c4-container.md`](05-c4-container.md) §3–§4. [`diagrams/c4-container.mmd`](diagrams/c4-container.mmd) draws the container/edge subset; the data-store and IdP/SOAR elements and the `F1`–`F11` flow labels are STRIDE elements named here, not nodes on that diagram. STRIDE letters apply per the Microsoft element-type mapping (§2); severity is qualitative Likelihood × Impact, and each row anchors to an NFR-SEC target with the per-zone framework cell from [`02-trust-boundaries.md`](02-trust-boundaries.md) §11.

Each threat names the actor it is reachable by. Primary adversary: A1, so the high-value rows weight the sandbox's outbound and host-facing edges, not the inbound caller edge.

| Actor | Who | Standing in this model |
|---|---|---|
| **A1** in-sandbox guest | the agent/code under execution, holding in-sandbox root | primary adversary — the model is built against it |
| **A2** external caller | an MCP-speaking caller, a data-plane client, or a network actor outside the perimeter | untrusted; authenticates at the gateway or the Web UI |
| **A3** host-foothold | a guest that already escaped the sandbox, or a compromised host / insider operator | a distinct trust level — an A3 threat is conditional on a prior escape (then it derives from the escape control, [NFR-SEC-02](manifesto/02-nfrs.md)) or sits behind the trusted-host assumption, not a failure of guest isolation |

## 2. Element inventory and STRIDE applicability

| Element | Type | Applicable STRIDE |
|---|---|---|
| P1 MCP gateway | process | S T R I D E |
| P2 Control / operator API | process | S T R I D E |
| P4-mount Object-store service + in-guest mount client (guest counterparty) | process | S T R I D E |
| P4-artifact Web UI (data-plane-client counterparty) | process | S T R I D E |
| P5 Session sandbox [1..N] | process | S T R I D E |
| P6 Egress trust-edge proxy | process | S T R I D E |
| P7 Audit pipeline | process | S T R I D E |
| D2 Audit pipeline store | data store (log) | T I D R |
| F1–F11 boundary flows | data flow | T I D (+ S/E per boundary) |
| E1 caller · E2 operator · E3 IdP · E4 SOAR · E5 data-plane client | external | S R |

The storage element is split by counterparty ([ADR-0015](adr/0015-storage-decomposition-by-trust-plane.md)). **P4-mount** is the guest counterparty: the in-guest mount client carrying the `filesystem_id`-scoped data leg, and the Object-store service it dials — the only door to the storage engine. **P4-artifact** is the Web UI fronting the external data-plane client (E5): the file HTTP API, the embeddable SPA, and preview-render ([components/08-web-ui.md](components/08-web-ui.md)). The two hold different aggregate roots (the session vs the artifact-plus-principal) and different blast radii, so their rows are enumerated separately. Neither holds a storage signing key. The signing key for the bearer is held off-box at a separate credential issuer ([ADR-0013](adr/0013-storage-credential-custody.md)); the issuer is not a modelled OCU container, so its own at-rest key exposure is the issuer operator's, outside this boundary.

Element IDs P3 and D1 are absent: they named the Credential custody process and its store, removed by [ADR-0005](adr/0005-egress-credential-delivery-envoy-sds.md). A configured upstream credential reaches P6 over Envoy SDS from an external source (a static file or a customer store); the surviving threats are carried on the P6 rows, and the SDS source's at-rest exposure is the source operator's, outside this boundary. The baseline egress hop injects no upstream credential and forwards the caller's own bearer unmodified ([ADR-0016](adr/0016-egress-baseline-inspection-hop-backend-scope.md)); edge-injection and the per-SNI inspection-leaf minter ride the optional hardening rung ([ADR-0007](adr/0007-egress-auth-mechanism.md)). The two SDS surfaces are distinct: the OCU per-SNI leaf-minter signs the inspection certificate; the upstream-credential SDS source delivers the upstream bearer. Neither holds the storage signing key.

E1, E2, and E4 carry their S/R threats inline (P1, P2 rows). E5 (the data-plane client) carries its S/R threats inline on the Web UI (the P4-artifact §3.2 rows on F11). E3 (the customer IdP) is a relying-party trust anchor whose own spoofing/repudiation is owned by the customer's identity estate, outside the OCU boundary; its failure surfaces here only as the F3 availability dependency (P2-D2).

## 3. Live threats

This section details the PARTIAL threats: a controlling NFR is specified, implementation tracked. Fully-mitigated threats are in §4. §3.1 carries what an in-sandbox guest (A1) reaches, including the P4-mount rows; §3.2 carries what only an external caller (A2) or a host-foothold (A3) reaches, including the P4-artifact rows.
